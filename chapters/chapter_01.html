2.2 File digestion

2.2.1 Removing .cif files

Upon researching and downloading the whole proteome of an organism from the Alphafold server, the Alphafold server will provide the proteome in .cif and .pdb files. In order to save space on the hard drive of the computer, the .cif-files are first removed.

# Set your directory path - Don't forget to replace organism to organism of interest!
dir_path <- "./Alphafold/Z.maize"

# List all .cif files
cif_files <- list.files(path = dir_path, 
                        pattern = "\\.cif.gz$", 
                        full.names = TRUE)

# Remove all .cif files 
file.remove(cif_files)

With the .cif-files removed, the remaining .pdb files are unpacked for further research.

2.2.2 Unzipping .pdb files

In order to optimally access the content of the .pdb files, the files are first unzipped. The gunzip has a setting that normally allows the zipped file to be removed upon unpacking. This can take a relatively long time. Alternatively, the unzipped files can be moved to a different file e.g. here called unzipped, after which the zipped files can be removed manually. This allows the operations to be split, reducing chances for the system to crash while computing.

# List all .gz files
gz_files <- list.files(path = dir_path,
                       pattern = "\\.gz$",
                       full.names = TRUE)

# unzipped_dir <- file.path(dir_path, "unzipped")
# if (!dir.exists(unzipped_dir)) dir.create(unzipped_dir)

# Loop and unpack
for (file in gz_files) {
  output_file <- file.path(
    dir_path,
    sub("\\.gz$", "", basename(file))
  )
  
  gunzip(file,
         destname = output_file,
         overwrite = TRUE,
         remove = TRUE)
}

2.3 Structural comparison

In this study, the structural comparison between a pair of proteins is based on a global RMSD-calculation, based on the C-alpha backbone of the proteins, through the use of the functions pdbaln() followed by atom.select() and rmsd(), all being from the bio3d package. The functions are enveloped in a larger function (calculate_rmsd) for high-throughput analysis of all protein-pairs. At the same time, this larger function will also retrieve the sequence similarity between the protein sequences within a pair, retrieved from the pdbaln() function, for visualizing the relation between the RMSD and sequence similarity later on.

2.3.1 Loading of the dataframe

A dataframe is set up containing the information coming from the Doubletrouble-database. Before loading the information of this database however, the EnsembL ID’s are converted into Uniprot ID’s. This conversion is necessary to link to the proteins to the .pdb structures gathered from the Alphafold database. Three columns are added to the dataframe for storing the results coming from calculate_rmsd function (see further): namely, the sequence identity, RMSD, and the B-factor after structural analysis is added per pair of pdb’s. The B-factor computation is applied to deduce wether there is confounding of the B-factor with the RMSD. In Alphafold, the B-factor column of the .pdb’s is applied to express Alphafold’s confidence in predicting the structure. The higher this result, the more confidence Alphafold has in its prediction.

# Read file coming from Doubletrouble database, in which the entries are converted from Ensembl to Uniprot database conversion
data <- read.delim("./Doubletrouble/zma_pairs_uniprot.tsv", stringsAsFactors = FALSE) |>
  # During the conversion of Ensembl to Uniprot, several duplicate pair entries were duplicated.
  # Through the distinct-function, these duplicate pairs of duplicates are filtered out to retrieve, to consider only one of these duplicates. As such, each pair of duplicates is considered once, which is ideal for statistical analysis 
  distinct() |>
  # During the conversion of Ensembl to Uniprot, several duplicate pair entries had instances where dup1 would be equal to dup2. These are filtered out. 
  filter(dup1 != dup2)
# Create empty numeric columns for storing the sequence identity (in seq_id), rmsd between pdb's (in rmsd) and average B-factor
data$seq_id <- NA_real_
data$rmsd <- NA_real_
data$bavg <- NA_real_

2.3.2 Computing RMSD- and sequence-similarity aswell as B-factor values for each pair of duplicates

The RMSD is computed using the rmsd() function from the bio3d library. This function requires a sequence alignment of the protein sequences, which can be performed by pdbaln(), coming from the same library. To perform the sequence alignment, the library requires an older version of MUSCLE (ver. muscle3.8.31), which can be downloaded at https://drive5.com/muscle/downloads_v3.html. For further installation instructions, consult the “installing Bio3D” section at the following link: http://thegrantlab.org/bio3d_v2/tutorials/installing-bio3d. From the pdbaln() function also allows extraction of the sequence similarity and the B‑factor values. Although the B‑factor is an inherent property of a .pdb file, the B‑factors extracted throughpdbaln() correspond only to the atoms that were used during the structural alignment. Considering we are comparing the global fold between two homologues protein structures, it has been opted to select for Calpha-atoms only. This reduces inflation of the RMSD-value coming from differences in side-chain positioning between two structures. Therefore, not all atoms from the original structures are included. Additionally, if the two protein sequences differ in length, certain residues will not participate in the alignment, further reducing the set of B‑factors returned.

# Specify function
get_values <- function(pdb1, pdb2){
# Specify the paths of both pdb's 
pdb1path <- paste0("C:/OneDrive/OneDrive - UGent/Academiejaren/jaar5_semester2/Masterproject/Data/Alphafold/Z.maize/AF-", pdb1, "-F1-model_v6.pdb")
pdb2path <- paste0("C:/OneDrive/OneDrive - UGent/Academiejaren/jaar5_semester2/Masterproject/Data/Alphafold/Z.maize/AF-", pdb2, "-F1-model_v6.pdb")
# Read PDB-file
# The ATOM.only = TRUE is useful for speed enhancements with large files where secondary structure, biological unit and other remark records are not required. This does not influence the output necessary for this experiment. 
pdb1 <- read.pdb(pdb1path, ATOM.only = TRUE)
pdb2 <- read.pdb(pdb2path, ATOM.only = TRUE)

#Select for C-alpha atoms 
ca1 <- atom.select(pdb1, "calpha")
ca2 <- atom.select(pdb2, "calpha")

# Perform alignment
aln <- pdbaln(list(pdb1, pdb2), fit = FALSE, atom.select = list(ca1, ca2))
# Retrieve sequence identity from alignment 
# The seqidentity-function returns a 2 x 2 matrix based on the alignment. One element is specified to be retrieved from this matrix that compares pdb1 vs. pdb2
id <- seqidentity(aln)[1, 2]

# pdbaln is a wrapper function that takes on multiple tasks. In order to relieve some of the computational load, the fitting will be performed within the RMSD-function. 
# If the fit-parameter within the rmsd-function is set to TRUE, the RMSD coordinate superposition is performed prior to RMSD calculation. As such, it is expected that the RMSD-function will take on to fit the sequences itself
# The RMSD-function returns a 2 x 2 matrix based on our alignment. One element is specified to be retrieved from this matrix that compares pdb1 vs. pdb2
rmsd_val <- rmsd(aln, fit=TRUE)[1, 2]

# Extract the B-factors per compared pair of atoms within the alignment 
# use apply() function across column to identify which column has two B-factor entries, meaning these were structurally aligned, from columns that contain only one B-factor entry, meaning the structures were not compared at that position
aligned_cols <- apply(aln$b, 2, function(col) all(!is.na(col)))
# Retrieve the B-factors of the positions that were structurally aligned and compared
aligned_scores <- aln$b[, aligned_cols]
# Get the average of the B-factors
bavg <- mean(aligned_scores)

# Return the computed RMSD- and sequence identity values in the form of a vector 
return(c(rmsd_val, id, bavg))
}

2.3.3 Iterating through data

The sequence similarity, retrieved from the pdbaln()function, and the RMSD-value, from the rmsd() function, are computed for each pair of duplicates and stored within the dataframe through a for-loop. The former two functions are enveloped in a larger function called calculate_rmsd(). If calculate_rmsd() fails to perform any of the former functions (e.g. due to the abscence of a UniProt structure within the Alphafold database), NA-values are set to replace any errors. This allows the function to continue and iterate over all the rows within the dataframe, without any errors interrupting the for-loop.

# Iterate through each row
for (row in 1:nrow(data)) {
  # At row x retrieve the name of the pdb-file at resp. column "dup1" and "dup2"
  pdb1 <- data[row, "dup1_uniprot"]
  pdb2 <- data[row, "dup2_uniprot"]
  
  # tryCatch enables to continue the iteration even though errors might occur
  vals <- tryCatch(
    # Execute expression; in this case, the function calculate_rmsd
    expr = {get_values(pdb1, pdb2)},
    
    # If an error occurs, return 'NA'
    error = function(e){return(NA)}
  )
  # The values obtained from the function are stored within the dataframe 
  data$rmsd[row] <- vals[1]
  data$seq_id[row] <- vals[2]
  data$bavg[row] <- vals[3] 
}

2.4 Visualization

With the data digested and values computed, this section aims to bring together the data into a concise visualization.

2.4.1 Data prepping

Before visualization, some tweaks still have to be made to the data in order to avoid getting warnings and to maintain consistency throughout the figures. For the former, the entries that failed to produce an RMSD-value, therefore containing an NA-value within the rmsd-column, are removed. As for the latter, the types of duplications are ordered, ensuring that the order throughout the figures remains the same.

data_ordered <- data |>
  # Remove homologues pairs that failed to compute an RMSD-value
  filter(!is.na(rmsd)) |>
  # Order the data based on type of duplication
  transform(type=factor(type,levels=c("SD","TD","PD", "rTRD", "dTRD", "DD")))

2.4.2 Plotting RMSD vs. type of duplication

To gain better insight on how the RMSD varies between the types of duplications, the RMSD is plotted as a violin plot per type of duplication. Besides that, the differences between groups is also statistically evaluated.

2.4.2.1 Statistical analysis

It is important to use an appropriate test, that has the most power, to which the data meets the required assumptions. Multiple transformations were attempted with the data, but repeatedly normality was not met. That is why the significance of the data will be investigated using a Kruskal-Wallis test. This test requires the presence of three or more groups within the data and expects the dependent variable to be at least on the ordinal scale. Considering the distributions differ in shape and variance (see figure p), the Kruskal-Wallis test will test differences in distributions of the RMSD between different groups.

kruskal_test(rmsd ~ type, data = data)

The p-value is less than the significance level 0.05, therefore there are significant differences between the types of duplications in terms of the distribution of RMSD.

2.4.2.2 Post-hoc testing

Considering the Kruskal-Wallis test is significant, the analysis continues with post-hoc testing. For this a Conover-Iman test will be applied. The Conover-Iman gains preference over the Dunn’s test for its increased power (without requiring additional assumptions), therefore being able to avoid increased probability of having a type II error. Given that the sample size of each group is at least n = 660, the dataset is expected to do well with this test.

conover <- conover.test(data$rmsd , data$type, method="holm")

Using Holm-Bonferroni to adjust the p-values, the Conover-Iman test found almost every comparison between two groups significant, except for the comparison between dTRD - rTRD and PD - TD.

2.4.2.3 Compact letter display

In order to provide a visual summary of the significance groupings across all pairwise comparisons, a compact letter display (CLD) is applied to assign letters that represent which groups are statistically indistinguishable.

# Generate a CLD using the cldList function from the rcompanion-library
cld <- cldList(comparison = conover$comparisons,
               p.value    = conover$P.adjusted,
               threshold  = 0.05) |>
  # Select relevant columns for the visual representation
  select(Group, Letter) |>
  # Capitalize the letter-notation
  mutate(Letter = toupper(Letter), pos_y = max(data_ordered$rmsd))

2.5 Functional enrichment analysis

The functional enrichment analysis consists of the analysis of the top 10% most divergent proteins and the top 10% least divergent proteins (based on RMSD-calculation) and is aimed to evaluate enriched biological processes, Reactome pathways, and protein domain categories.

2.5.1 Data preparation

The first step is to index the data based the sorted RMSD-value and remove entries where there’s no RMSD-value (rmsd = NA).

# Index the data based on rmsd-value
data_idx <- data_ordered |>
  # Sort based on rmsd 
  arrange(rmsd) |>
  # Index based on sorting based on rmsd 
  mutate(idx = row_number())

2.5.2 Administer STRING-ID’s

The STRING database is applied to denote the biological process, reactome pathways and protein domains. The database makes use of STRING-ID’s to streamline the ID’s coming from different databases. The name of the protein there has to be linked to the protein STRING-id within the database. This conversion is mediated through the protein aliases dataframe, acquired from the STRING database.

# Load STRING protein aliases
df_aliases <- read_tsv(
  "https://stringdb-downloads.org/download/protein.aliases.v12.0/4577.protein.aliases.v12.0.txt.gz",
  show_col_types = FALSE) |>
  # Subset aliases from STRING protein aliases file 
  subset(select = c("#string_protein_id", "alias")) 

# Rename columns 
colnames(df_aliases) <- c("string_id", "alias")

# Left_join() from the dplyr-library is applied to match the protein name in data with the STRING-id stored within the database. This is applied to both duplicates, starting with dup1
data_idx <- left_join(data_idx, df_aliases, by = c("dup1_uniprot" = "alias")) |> 
  # Left_join appears to duplicate the entries, therefore; apply distinct() to select for distinct rows
  distinct() |> 
  # Rename the recently added column 
  rename("string_id_dup1" = "string_id")

# The same is applied to dup2
data_idx <- left_join(data_idx, df_aliases, by = c("dup2_uniprot" = "alias")) |>
  distinct() |>
  rename("string_id_dup2" = "string_id")

# df_aliases is no longer required, in order to have a clean Workspace, it is advised to remove it
rm(df_aliases)

2.5.6 Enrichment analysis

With the top 10% lowest and highest scoring entries based on RMSD-value specified in separate dataframes and the enrichment terms specified ready for analysis, functional enrichment can be performed.

2.6 B factor computation

Based on visual analysis of two compared structures, there was a hypothesis that the confidence of Alphafold may confound the degree of divergence (based on RMSD) between structures. To test this hypothesis, the B-factor was previously extracted and stored in a dataframe, along with the sequence similarity and RMSD, when running the get_values function.

2.6.1 Alternative average B-factor computation

The B-factor computed during the get_values function is the average B-factor per pair of homologous, at the positions where both structures were aligned and therefore compared. In order to confirm that this is indeed the B-factor, the confidence score is also computed via an alternative way. Here, the confidence score is extracted, per structure, from the Alphafold server via API. Considering the computation of the average B-factor at positions where only both of the sequences is aligned is a rather intensive process, it would also be interesting to check whether this type of computation makes a difference relative to simply extracting the confidence score from the Alphafold server.

In this step, a function called get_average_plddt is set up. This function extracts the confidence score from the Alphafold website per protein ID.

# Define function to extract average pLDDT score from AlphaFoldDB
get_average_plddt <- function(protein_id) {
    # Per protein_id, retrieve the URL to the page of the protein
    url <- paste0('https://alphafold.ebi.ac.uk/api/prediction/', protein_id)
    # Using the httr-library, extract the contents from the URL and retrieve the confidence score (globalMetricValue)
    score <- httr::content(httr::GET(url))[[1]]$globalMetricValue
    return(score)
}

With the function specified, the confidence score is extracted for every protein ID considered in previous datasets.

# Get all protein entries within the dataset
proteins <- unique(c(data_ordered$dup1_uniprot, data_ordered$dup2_uniprot))

# Per protein entry, get the confidence from the Alphafold website 
plddt_df <- lapply(proteins, function(x) {
  score <- tryCatch(
    # Execute expression; in this case, the function get_average_plddt, which is specified earlier
    expr = {get_average_plddt(x)},
    # If an error occurs, return 'NA'
    error = function(e){return(NA)}
  )
  # Create a dataframe with the protein entry and their respective confidence score, retrieved from the Alphafold website
  df <- data.frame(id = x, score = score)
}) |>
  # Combine the set of dataframes to form a bigger dataframe 
  bind_rows()
